# 推广与验收

英文版：[../../practice/07-rollout-and-acceptance.md](../../practice/07-rollout-and-acceptance.md)

## Phase 1：试点，Weeks 1-4

范围：

- 选择一个中等复杂度业务领域，例如账户管理、流程处理、审批流或服务集成。
- 包含一个内部团队，采用 Superpowers 工作流。
- 如果供应商输出属于试点领域，只按交付物验收方式纳入。

交付物：

- SDD Story 规格 模板。
- 技术规格 模板。
- 测试规格 模板。
- Prompt Card 模板。
- MR 模板。
- 质量门禁 checklist。
- 初始 metric dashboard 定义。
- Superpowers 轻量、标准和高风险 工作流 定义。

目标：

- 验证流程是否可运行。
- 此阶段不要优先追求完整效率提升。

## Phase 2：扩展，Weeks 5-8

范围：

- 扩展到所有团队。
- 覆盖更多服务模块和接口变更。

交付物：

- 批准后的知识库。
- API contract repository。
- Test asset repository。
- SonarQube、SCA、Secret Scan 和 artifact repository 集成。
- 每周 AI-SDD quality 评审。

目标：

- 让 SDD 和基于 Superpowers 的 AI 治理成为内部默认交付工作流。

## Phase 3：规模化，Weeks 9-12

范围：

- 建立 developer portal 和 module 负责人hip map。
- 将供应商交付质量纳入供应商管理。

交付物：

- Service catalog。
- 模块负责人 registry。
- Dependency map。
- AI usage dashboard。
- Quality and consistency dashboard。
- 供应商质量评分卡。

目标：

- 从流程采用进入可度量交付系统。

## 试点验收场景

相同验收标准，不同交付方式：

- 内部团队使用 Superpowers 工作流，供应商工作只按交付物评估。
- 检查两种输出是否满足同一 SDD Spec、API contract、测试证据和质量门禁期望。

AI MR 审计：

- 随机抽样 10 个 AI 辅助 MR。
- 确认每个 MR 都有需求链接、SDD Spec、AI usage declaration、测试证据和人工评审。

核心 API 变更：

- 修改一个核心接口。
- 验证 OpenAPI、Mock、测试和调用方影响分析已更新。

低质量 AI 输出：

- 在受控分支提交一个刻意不完整的 AI 生成实现。
- 验证质量门禁会阻止合入。

新成员启用：

- 让一名新员工用轻量 Superpowers flow 完成低风险 Story。
- 验证其在有限辅导下可以完成内部工作流。

缺陷追溯：

- 选择一个缺陷，追溯到需求、SDD Spec、代码、测试、评审和发布。

## 项目级成功标准

- 所有团队使用相同 SDD 工件模板。
- 内部团队使用约定 Superpowers 工作流 tiers。
- 100% 核心模块有 Owner。
- 100% 新服务 API 有 OpenAPI 契约。
- 100% MR 通过自动化质量门禁。
- 100% AI 辅助代码有人工评审和审计记录。
- 试点 cycle time 改善，且 defect escape rate 不上升。
