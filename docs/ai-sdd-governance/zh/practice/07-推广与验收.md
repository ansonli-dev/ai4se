# 推广与验收

英文版：[../../practice/07-rollout-and-acceptance.md](../../practice/07-rollout-and-acceptance.md)

## 目的

本篇负责 AI-SDD 推广的**验收**——验证 [优先级与路线图](06-优先级与路线图.md) 规划的推广是否真的产生了你想要的行为变化的具体场景和项目级标准。

本篇刻意不维护自己的推广阶段。[优先级与路线图](06-优先级与路线图.md) 是 5 阶段（Phase 0 → Phase 4）推广序列的正典；本篇验证每个阶段的结果。

## 与路线图的关系

```text
优先级与路线图（doc 06）                推广与验收（doc 07）
──────────────────────────              ──────────────────────────
Phase 0：方向与最小规则                  → Phase 1 开始时这些场景就适用。
Phase 1：以手工为主的受控试点            → 试点场景（本篇）验证受控试点。
Phase 2：CI 和质量 baseline              → 质量门禁场景验证门禁真的会阻断。
Phase 3：多团队扩展                      → 跨团队场景验证规模化采用。
Phase 4：平台和 Dashboard 投入           → 项目级标准验证可度量系统。
```

如果 doc 06 中某个 Phase X 被声称"完成"但本篇对应的验收场景没通过，那这个阶段实际上没完成。

## 试点验收场景

下面六个场景是 AI-SDD 推广的核心验收测试。每一个都验证一个具体的治理属性是真实的、不是口号。

### 相同验收标准，不同交付方式

测什么：内部 AI 辅助工作和供应商仅交付物的工作虽然用不同工作流，但被同样的产出标准衡量。

怎么跑：

- 在同一个迭代里挑一个走 Superpowers 工作流的内部 Story 和一个供应商交付物。
- 检查两种输出是否满足同一 SDD Spec、API contract、测试证据和质量门禁期望。
- 验证验收按交付物证据决定，不按由哪种工作流生产决定。

通过标准：两个按同一标准要么都通过、要么都不通过。

### AI MR 审计

测什么：AI 辅助 MR 是可审计的。

怎么跑：

- 从最近迭代里随机抽样 10 个 AI 辅助 MR。
- 确认每个都包含：需求链接、SDD Spec 链接、AI usage declaration、测试证据、人工评审。

通过标准：10 / 10 全部包含五项。

### 核心 API 变更

测什么：核心接口变更时契约纪律是否成立。

怎么跑：

- 把一个核心接口变更作为计划内 Story。
- 验证 OpenAPI 已更新、mock 已更新、契约测试已更新、调用方影响分析已记录。

通过标准：没有契约测试缺失；没有调用方对变更感到意外。

### 低质量 AI 输出

测什么：质量门禁是否真的拦得住坏输出。

怎么跑：

- 在受控分支提交一个刻意不完整或浅层的 AI 生成实现。
- 验证质量门禁阻止合入，评审者清单也抓到门禁漏掉的部分。

通过标准：这个坏变更通过任何路径都合不进去。

### 新成员启用

测什么：工作流对新人是否可教。

怎么跑：

- 让一名新员工用轻量 Superpowers flow 完成一个低风险 Story。
- 验证其在有限辅导下能完成内部工作流。

通过标准：新人不需要私下例外规则、不需要无文档手把手就能完成 Story。

### 缺陷追溯

测什么：一个缺陷是否能在 [执行栈](../knowledge/03-执行栈.md) 的每一层都被追溯到。

怎么跑：

- 选一个缺陷（真实的或植入的）。
- 把它追溯回需求、SDD Spec、代码、测试、评审和发布。
- 跑 doc 03 的自底向上诊断：CI？Harness？Superpowers？SDD？

通过标准：能说出让这个缺陷漏过去的那一层，且下一迭代的改进目标清楚。

## 项目级成功标准

这些是 [优先级与路线图](06-优先级与路线图.md) 整个推广完成后的稳态标准。它们不是单个 Story 的门禁；它们是项目健康指标。

- 三个或以上内部团队使用相同 SDD 工件模板。
- 内部团队使用约定 Superpowers 工作流 tiers。
- 100% 核心模块在 `CODEOWNERS` 中有 owner。
- 100% 新服务 API 有 OpenAPI 契约。
- 100% MR 在合入前通过自动化质量门禁。
- 100% AI 辅助代码有人工评审和审计记录。
- 试点 cycle time 改善，且 defect escape rate 不上升。

这些标准也是 [指标](../knowledge/10-指标.md) 进入持续治理评审的输入。

## 什么时候跑验收

- **Phase 1（以手工为主的受控试点）结束时：** 跑 *相同验收标准*、*AI MR 审计*、*新成员启用* 三个场景。其他三个需要 CI/Phase 2 的成熟度才有意义。
- **Phase 2（CI 和质量 baseline）结束时：** 加 *低质量 AI 输出* 和 *核心 API 变更*。*缺陷追溯* 需要真实缺陷周期发生过才有意义。
- **Phase 3（多团队扩展）结束时：** 在至少两个团队跨团队跑完六个场景。开始把项目级标准当趋势跟踪，不当通过/不通过。
- **Phase 4 之后：** 验收场景变成回归检查——每季度跑一次，或者治理重大变化后跑。

## 要点回顾

- 本篇负责验收；doc 06 负责推广阶段。
- 六个试点场景验证那些必须是真实而不是口号的治理属性。
- 项目级标准是稳态健康指标，不是单 Story 门禁。
- 验收场景没通过之前，阶段不算完成。
