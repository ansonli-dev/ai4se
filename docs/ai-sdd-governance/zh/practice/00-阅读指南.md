# 实践阅读指南

英文版：[../../practice/00-reading-guide.md](../../practice/00-reading-guide.md)

## 目的

这是实践路径的第一篇。它给你一张图，说明七篇实践文档怎么互相配合、不同角色按什么顺序读、哪一篇是其他所有文档都引用的**中心参考**。

读完本篇你应该知道：

- 实践相对知识多做了什么。
- "我在阶段 X 需要什么工件？"这个问题的正典答案在哪一篇。
- 你的角色先读哪一篇。
- 知识路径里的四层执行栈在实践里怎么重现。

## 实践相对知识多做了什么

知识教模型，实践把它落地。[执行栈](../knowledge/03-执行栈.md) 的四层——SDD、Superpowers、Harness、CI/Review——在实践里仍是底层模型，但现在它们被接入团队真实的 SDLC 阶段、绑定到具体工件和角色、按真实时间线推广。

如果你读完了 [总结](../knowledge/11-概念到交付.md)，就已经看过一个 Story 走完整个栈。实践给你的是"怎么每天、按规模、跨多个团队和供应商持续这样做"的操作手册。

## 十篇文档一页看完

```text
00 阅读指南                  ← 你在这里
01 团队级 AI SDLC            ← 把栈映射到 SDLC 阶段；Superpowers / GSD / gstack 各放哪
02 AI 上下文工件地图          ★ 正典参考：阶段 × 工件 × Tier × 模板
03 Superpowers 采用策略       ← Tier A/B/C 规则和采用边界
04 开发者指南                 ← 日常工作流：Story 接收 → 已验证 MR
05 实施 Playbook              ← 推广运营：Week 0、Kickoff、RACI、仓库设置
06 优先级与路线图             ← P0/P1/P2 分类、5 阶段路线图、Backlog
07 推广与验收                 ← 验证推广真的落了地的验收场景
08 角色 × 阶段矩阵            ★ 正典参考：每个阶段谁产出什么
09 BA 指南                    ← BA 的深度工作流：需求 → 三道评审 → Story → 关闭
```

三个结构性要点要记住：

- **第 02 篇（AI 上下文工件地图）是工件参考的正典。** 其他每一篇实践文档关于"某阶段或某 Tier 需要什么工件"都指向它。
- **第 08 篇（角色 × 阶段矩阵）是角色参考的正典。** 它是第 02 篇的姊妹篇——工件 vs 角色。对角色深度工作流，第 08 篇链到独立指南（目前是 04 开发者指南 和 09 BA 指南）。
- **第 06 篇（路线图）是推广序列的正典。** 第 07 篇是"验证第 06 篇的推广确实落了地"的验收场景，它自己不维护单独的路线图。

## 按角色的阅读顺序

读完 00 之后，其他几篇大体独立、可按需回查。各角色的初次阅读顺序：

- **第一次完整阅读（所有人）**：00 → 01 → 02 → 03 → 04 → 08 → 09 → 05 → 06 → 07。先在工件地图后面读 08、09，看"谁产出什么"；剩下的是运营和推广。
- **Delivery Owner**：00 → 01 → 08 → 05 → 06 → 07。除非你同时也带工程或 BA，可以跳过 03/04/09。
- **架构师 / Tech Lead**：00 → 01 → 02 → 03 → 04 → 08，做路线图时加 06。读 09 知道 BA 欠你什么。
- **开发者**：00 → 03 → 04 → 08（你那一行），凡是"我现在需要什么工件？"就回 02 查。
- **BA**：00 → 02 → **09（你的深度指南）**→ 08（你那一行）→ 05 → 07。需求 → 三道评审 → Story 是你的主场。
- **PO**：00 → 01 → 09（读它知道 BA 需要你给什么）→ 08 → 06 → 07。
- **QA**：00 → 02 → 04 → 08（你那一行）→ 07，加 [测试策略](../knowledge/06-测试策略.md) 和 [Testing Policy](../../../ai/testing-policy.md)。
- **安全负责人**：00 → 02 → 08（你那一行）→ 05 → 07，加 [Security Policy](../../../ai/security-policy.md)。
- **AI Champion**：00 → 03 → 04 → 05 → 08（你那一行），把 02 当工件参考。
- **Module Owner**：00 → 08（你那一行）→ 02（看所有权模块的工件）。主要是评审 / 批准角色。
- **供应商交付负责人**：00 → 01 → 02（只读和交付物相关的阶段）→ 05（供应商评审章节）→ 07。

每一篇结尾都有 **要点回顾** 和 **下一篇** 引导。

## 从知识带过来的术语

实践默认你已经熟悉知识路径的术语（SDD、Superpowers、Harness、Tier A/B/C、MR、Owner Review、AI Champion）。一个实践专有的术语：

- **Context Package（上下文包）**——从一个交付阶段传给下一阶段的命名工件束，让下一阶段（包括 AI Agent）不必猜测。第 02 篇列了八种：Project、Architecture、Requirement Breakdown、Story、Implementation Evidence、Story Acceptance、Release Readiness、UAT And Feedback。

第 01 篇会引入两个相邻框架（那里有定义）：

- **GSD**——Get Shit Done。一种上下文工程和"规格驱动长任务执行"模式；用来包裹多阶段、有上下文腐烂风险的工作。
- **gstack**——一个角色驱动的交付循环：产品框定、设计评审、浏览器 QA、发布清单；按需用于发现、Web QA、发布纪律。

## 要点回顾

- 实践把知识路径同一个四层栈落地，不引入新的心智模型。
- 第 02 篇（工件地图）是"阶段 × 工件 × Tier"的正典参考；其他文档都指向它。
- 第 06 篇掌握路线图；第 07 篇掌握验证路线图真的落了地的验收场景。
- 按角色阅读顺序能省时间——多数读者首次阅读不需要把七篇全读完。
