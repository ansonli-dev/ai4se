# BA 指南

英文版：[../../practice/09-ba-guide.md](../../practice/09-ba-guide.md)

## 目的

本篇是 BA 的深度工作流指南——和 [开发者指南](04-开发者指南.md) 平行的 BA 角色版本。覆盖从原始业务请求到反馈闭环的完整 11 步 BA 工作流，以及每一步必须产出的工件和 AI 可用性条件。

如果你不是 BA，仍可能需要本篇来知道 BA 欠你什么。[角色 × 阶段矩阵](08-角色阶段矩阵.md) 给每个阶段的"谁做什么"概览；本篇是矩阵里 BA 那一行的深度展开。

## BA 在哪里

BA 的重心在 [执行栈](../knowledge/03-执行栈.md) 的**第 1 层（SDD）**——把业务意图转化为可评审的规格。但 BA 工作贯穿整个生命周期：回答开发者问题、把 Story 验收回链到父 Requirement、捕获反馈到下一迭代。

在 [角色 × 阶段矩阵](08-角色阶段矩阵.md) 中，BA 在八个阶段里活跃于七个（S0~S7，只在 S1 架构设计中休眠）。BA **端到端拥有**的两个阶段是：

- **S2——需求 → 三道评审 → Story 拆分**
- **S3——Story Ready → Backlog → Sprint 选取**

本指南给出这两个阶段的深度微流程，加上 BA 在其他阶段的参与方式。

## 术语说明

本手册中：

- **需求 / Requirement**——客户或业务 stakeholder 提出的 Feature 量级业务单元，记录在 [Feature Spec](../../../templates/feature-spec.md)（该模板含有此别名说明）。
- **三道评审**——每个 Requirement 的门禁评审：**需求评审**、**技术评审**、**测试评审**。合并证据在 [Requirement Review Record](../../../templates/requirement-review-record.md)。
- **Story**——拆分 Requirement 得到的产物；通过 [BA Handoff Checklist](../../../templates/ba-handoff-checklist.md) 签字后进入 Backlog。

一个 Requirement 大致**对应一个 Feature ≈ 多个 Stories**。三道评审 gate 的是 *Requirement*，不是单个 Story。

## BA 的 11 步工作流

流程总览：

```text
S2 范围：
  1. 需求接收
  2. 需求分析
  3. 需求评审  （Review Record §1）
  4. 技术评审  （Review Record §2）——与 Tech Lead 联合主持
  5. 测试评审  （Review Record §3）——与 QA 联合主持
  6. Story 拆分

S3 范围：
  7. Backlog 入库（带 Handoff Checklist）
  8. Sprint 计划支持

S4-S7 范围：
  9. 开发支持
  10. Story 验收（回链 Requirement）
  11. 反馈闭环（关闭 Requirement、提出更新）
```

下面每一步都有：**输入**、**步骤**、**产出**、**AI 可用性**（产出必须满足什么条件，下游 AI 辅助工作才能安全进行）、**陷阱**。

### Step 1：需求接收

**输入：**
- 业务请求（工单、对话、stakeholder 邮件、升级）。
- 原始 stakeholder 身份和角色。
- Project / Iteration Brief（范围上下文）。

**步骤：**
1. 原样捕获原始请求，不要改写。
2. 分配 Requirement ID。
3. 识别真正的原始 stakeholder（不是转发的人）。
4. 标注紧急度、deadline、升级来源。
5. 放进 BA 队列，加"raw / 未分析"标签。

**产出：**
- BA 队列里的新 Requirement 条目：ID、原始请求、原始人、接收日期、紧急度。

**AI 可用性：**
- Requirement ID 是所有下游工件回链的脊柱——确保它从第零天就唯一、可见。

**陷阱：**
- 在捕获之前改写请求（你失去原话，这在 stakeholder 争议时很重要）。
- 在还没决定是否在范围内之前就把 Requirement 分给自己——接收和分析要分开。

### Step 2：需求分析

**输入：**
- Step 1 的原始 Requirement。
- Project / Iteration Brief。
- 现有相关 Feature Spec、Architecture Constraints、可能重叠的历史 Requirements。

**步骤：**
1. 起草 [Feature Spec / Requirement Spec](../../../templates/feature-spec.md)：
   - 业务目标：一句话。
   - 用户和用例：谁，做什么，预期什么结果。
   - 功能范围：in / out 列表。
   - 业务规则：显式规则表。
   - 用户流：至少主流程（Mermaid 或编号列表）。
   - 数据和接口影响：数据对象、API、事件、集成、错误码。
   - 非功能需求：权限、审计、性能、兼容性、安全、可观测性。
   - Story 拆分（初步草稿——评审后再确认）。
   - 验收策略：自动化、人工、是否需 UAT、是否需集成测试。
2. 识别明显依赖（其他团队、供应商、环境）。
3. 标出你自己回答不了的问题——让它们以问题形式可见，不要变成假设。
4. 发给 Tech Lead + QA + PO 做预审（评审准备度检查）。

**产出：**
- `Status: Ready For Review` 的 Feature Spec 草稿。
- 需求评审上要抛出的 open question 列表。

**AI 可用性：**
- Feature Spec 必须写出真实用户、真实数据对象、真实 API——不是"相关的"或"受影响的系统"。AI agent 以后读它无法把"受影响的系统"解析成任何具体的东西。
- 非功能需求要具体（如"审计日志必须捕获用户 ID、时间戳、尝试动作、结果"），不是泛化的（"可审计"）。

**陷阱：**
- 用自己的答案填模糊业务问题，而不是把它们标成问题。
- 把假设埋在散文里（"用户大概想要 X"）——假设应该放在自己的标记列表里，让评审者挑战。

### Step 3：需求评审

**输入：**
- Step 2 的 Feature Spec 草稿。
- Open question 列表。
- Stakeholder 的可用时间。

**步骤：**
1. 安排评审。必需出席：PO、BA（主持）、原始 stakeholder。被咨询：Delivery Owner。
2. 带着会议室走一遍 Feature Spec。
3. 把每个问题和决议捕获到 [Requirement Review Record](../../../templates/requirement-review-record.md) 的 §1。
4. 推动得到显式决策：确认范围、确认非范围、成功标准、约束、假设。
5. 决定结果：
   - **Pass**——进入技术评审。
   - **Pass with conditions**——进入下一步但 conditions 不可让步。
   - **Fail**——返回 Step 2 重做。

**产出：**
- Review Record §1 完成，讨论日志四列都填好，确认的范围/非范围/成功标准、行动项、结果。

**AI 可用性：**
- "确认范围"必须是能力列表，不是段落。下游读 Story 的 AI agent 需要对照离散的范围项。
- 每个"Pass with conditions"的 condition 必须有 owner 和验证方式——否则 condition 只是装饰。

**陷阱：**
- 把评审当成讲解而不是工作会议——如果没人挑战任何东西，要么规格平庸，要么屋里的人不对。
- 让"以后再想"过关——把它们捕获成带 owner 和日期的行动项。

### Step 4：技术评审——与 Tech Lead 联合主持

**输入：**
- §1 结果（必须 Pass 或 Pass with conditions）。
- Feature Spec。
- 受影响模块的 Architecture Constraints。
- 可能受影响的 ADR。

**步骤：**
1. Tech Lead 主持；BA 参与，确保业务意图在技术重述中不丢失。
2. 必需出席：Tech Lead（主持）、BA、受影响模块的 Module Owner。安全敏感时必需：Security Lead。被咨询：开发者代表。
3. 走一遍技术影响：受影响模块、API、事件、数据模型、权限、外部集成。
4. 识别需要的 ADR；如果有架构决策待定，本 Requirement 不能完成此评审。
5. 估算复杂度和顶级风险；约定 mitigation。
6. 把以上都捕获到 Review Record §2。
7. 决定结果：Pass / Pass with conditions / Fail（返回重设计）。

**产出：**
- Review Record §2 完成。
- 必要时 ADR 草稿。
- 如果技术范围回灌到规格，则更新 Feature Spec。

**AI 可用性：**
- 技术影响列表必须写出真实文件路径或模块名，不是类别。AI agent 后面需要定位它们。
- 风险 mitigation 必须是行动，不是愿望（"在 X 模块加 circuit breaker"，不是"我们会小心级联失败"）。

**陷阱：**
- BA 退后让对话漂离业务意图——你的工作是在屋里开始争实现时把业务需要解读出来。
- 涉及 auth/data/payment/audit 时漏掉 Security Lead 参与。

### Step 5：测试评审——与 QA 联合主持

**输入：**
- Review Record §1 和 §2。
- 受影响模块的现有 Test Spec 模式。
- 现有 Acceptance Strategy 模板。

**步骤：**
1. QA 主持；BA 参与，在 AC 模糊时 push back。
2. 必需出席：QA Lead（主持）、BA、Tech Lead。安全敏感时必需：Security Lead。
3. 确认 AC 是 Given/When/Then 形式、且按字面可测试。
4. 识别必需测试层级：unit / component / contract / integration / E2E。
5. 识别回归覆盖、权限测试、审计日志测试。
6. 确认测试环境和数据需求（涉及客户数据时含脱敏方案）。
7. 定义 UAT 范围。
8. 把以上都捕获到 Review Record §3。
9. 决定结果：Pass / Pass with conditions / Fail（返回 AC 重写）。

**产出：**
- Review Record §3 完成。
- 如果 AC 不可测试，反馈回 Step 2 重写。

**AI 可用性：**
- "按字面可测试"意味着 AI agent 以后生成测试时能直接从 AC 文本产出断言，不需要编造细节。
- 必需测试层级显式，意味着 agent 知道写什么测试，而不是"加点测试"。

**陷阱：**
- 接受 QA 能理解但开发者直接测不了的 AC。
- 跳过"AI 生成测试风险"检查——如果测试方案依赖大量 mock，现在就标出来，不要拖到 MR 评审。

### Step 6：Story 拆分

**输入：**
- 三个评审节都 Pass（或 Pass with conditions）。
- 批准的 Feature Spec。

**步骤：**
1. 重读 Feature Spec 的 Story 拆分草稿。
2. 和 Tech Lead、QA 一起把 Requirement 拆成 Stories，使每个 Story：
   - 独立可验证一个行为（或显式依赖排序，且原因写下来）。
   - Tier A 适合一个开发者会话，Tier B/C 适合一个 sprint。
   - 有单一 Module Owner（或明确的主要 owner）。
3. 对每个 Story：
   - 创建 [Story Card](../../../templates/story-card.md)：目标、角色、AC、父 Requirement 链接。
   - 分配 Tier（A / B / C）及理由。
   - 识别 Module Owner。
   - 列出初始依赖。
4. 在 Review Record 的 Traceability 表中记录追溯。

**产出：**
- 每个拆出的 Story 的初始 Story Card。
- Review Record 的 Traceability 表，链接每个 Story 到父 Requirement。

**AI 可用性：**
- 每个 Story 有到父 Requirement 和 Review Record 的反向链接——agent 拿到 Story 就能追溯上游上下文。
- Story 大小匹配 Tier——一个本来是 Tier A 但需要 ADR 的 Story 是被错分类的 Tier C。

**陷阱：**
- 把一个 Story 切给两个 Module Owner 而没有主 owner——这个 Story 会停摆，因为没人觉得自己负责。
- "技术分解"型 Story（"建表"、"加路由"），它们不独立产生业务价值——这种应当合进一个业务价值的 Story 里，而不是作为兄弟 Story。

### Step 7：Backlog 入库

**输入：**
- Step 6 的初始 Story Cards。

**步骤：**
1. 对每个 Story 完成 [BA Handoff Checklist](../../../templates/ba-handoff-checklist.md)：
   - 来源（父 Requirement 追溯）。
   - Story Card 完整性（目标、角色、流程、AC）。
   - AI Context Boundary（允许/禁止上下文，可编辑/禁止范围，验证命令）。
   - 链接工件（SDD Story Spec、Technical Spec、Test Spec、契约等，按 Tier）。
   - 依赖和排序。
   - Open questions 为空。
2. 为 Tier B/C 编写或更新 [SDD Story Spec](../../../templates/sdd-story-spec.md)。
3. 获得签字：
   - BA 签字（始终）。
   - Tech Lead 签字（Tier B/C）。
   - QA 签字（Tier B/C）。
   - Module Owner 知会（Tier C）。
4. 把 Story 放进 Backlog，标好优先级和依赖。
5. 跑清单上的 AI 可用性自测；任一答案"否"就把 Story 暂停而不是入库。

**产出：**
- Backlog 中的 Story，带 Handoff Checklist 链接。
- SDD Story Spec 链接到位。

**AI 可用性：**
- 清单的 AI 可用性自测通过——这是"Story 已可被 AI 辅助执行"的契约条件。

**陷阱：**
- 把清单当成走流程——它的目的是 *fail closed*。任何未解决的问题留在父 Requirement，永远不要悄悄落到 Story 上。
- 为了"可见性"把 Story 放进 Backlog——Backlog 中存在就意味着已 ready。

### Step 8：Sprint 计划支持

**输入：**
- 含 ready Stories 的 Backlog。
- PO 的优先级。
- Delivery Owner 给的团队容量。

**步骤：**
1. 带 Sprint Planning 走候选 Stories。
2. 当场回答依赖 / 范围问题；如果某问题需要回到父 Requirement，那个 Story **不** ready 进本 sprint。
3. 确认 Tier C 入选有 Module Owner 和 Delivery Owner 签字。
4. Sprint 确认后冻结本 sprint Story 集；中途变更走 [Change Request](../../../templates/change-request.md) 流程。

**产出：**
- 确认的 Sprint 范围写入 [Iteration Brief](../../../templates/iteration-brief.md)。
- 任何被拉回 Backlog 的 Story 及原因。

**AI 可用性：**
- 确认的 Sprint 范围是 agent 和人共同的"本 sprint 包含什么"权威来源——确保在单一权威位置。

**陷阱：**
- Sprint 中途对一个本来没 ready 的 Story 说"行"——这就是 Requirement 级问题最后在 Story 评论里争论的原因。

### Step 9：开发支持

**输入：**
- S4 期间开发者的问题。

**步骤：**
1. 分诊问题：
   - Story 级澄清 → 在 Story 上回答，更新 SDD Story Spec。
   - Requirement 级澄清 → 更新父 Requirement 和 Review Record；级联到所有子 Story；在 BA 队列标记变更。
   - 范围外请求 → 拒绝；建议走 Change Request。
2. 更新工件，不在聊天里答。任何应该流到下一个 agent 会话的东西都必须落进文件。
3. 如果澄清暴露了 Review Record 层面的缺口（如 AC 本来就不可测试），触发针对性的重评审——不要光打补丁在 Story 上。

**产出：**
- 带时间戳的澄清更新到 Story Card / SDD Story Spec。
- 如果范围变了，更新 Requirement 和 Review Record。

**AI 可用性：**
- 澄清写进 agent 会读的同一份工件——跨 agent 会话保留。
- 父 Requirement 上的任何范围变化都传播到所有子 Story，不只是 BA 自己脑子里记着。

**陷阱：**
- 在 Slack 答完不更新规格——下一个开发者（或 AI agent）看不到。
- 一个 Story 的澄清悄悄改变了兄弟 Story 的隐含范围。

### Step 10：Story 验收——回链到 Requirement

**输入：**
- Story 完成时的 Story Acceptance Record。

**步骤：**
1. 每个验收的 Story，更新父 Requirement：链到 Story Acceptance Record、日期、任何记录的问题。
2. 当一个 Requirement 的所有子 Story 都验收，把 Requirement 标为 `Status: Accepted`（如有条件保留则 Conditionally Accepted）。
3. 如果某个 Story 被拒收，决定：
   - Story 级返工（最常见）→ 回到 S4。
   - Requirement 级返工（少见但真实）→ 提 Requirement 更新并触发部分重评审。
4. 适当时与 PO 一起参与业务验收。

**产出：**
- 更新的父 Requirement 状态。
- 在 Story 和 Requirement 两个级别链接的验收证据。

**AI 可用性：**
- Requirement 级状态反映累积的 Story 验收——未来 agent 或 stakeholder 问"Requirement X 真的发布了吗"能得到真实答案。

**陷阱：**
- 一个 Story 还在返工就把 Requirement 标为 Accepted。
- 把"Story 合入"和"Story 验收"等同——验收来自 PO/QA/业务，不来自合入。

### Step 11：反馈闭环——关闭 Requirement

**输入：**
- S7 的 UAT 结果。
- 追溯到本 Requirement 的生产事件。
- 交付后的 stakeholder 反馈。

**步骤：**
1. 显式收集反馈——不要等它从日常对话里飘过来。
2. 每个反馈分类：
   - **缺陷** → 记录缺陷，按 [Defect Attribution](../../../templates/defect-attribution.md) 归因，触发修复 Story。
   - **变更请求** → 提 [Change Request](../../../templates/change-request.md)；它在下迭代成为新（或修订）的 Requirement。
   - **知识库更新** → 当公认的业务知识变化时，提 [Knowledge Base Update](../../../templates/knowledge-base-update.md)。
   - **表扬 / 无 op** → 简短记录；无后续动作。
3. 关闭 Requirement：状态、经验教训、链到 follow-up Requirement（如有）。
4. 在 [Weekly AI-SDD Review](../../../templates/weekly-ai-sdd-review.md) 上暴露 BA 流程改进点（如反复出现的 AC 模糊类型）。

**产出：**
- 已关闭的 Requirement 带经验教训和 follow-up 链接。
- 已提的 Change Request / Knowledge Base Update。
- 周度复盘中的 BA 流程改进项。

**AI 可用性：**
- 带经验教训关闭的 Requirement 成为下迭代 BA 工作的结构化上游上下文——失败模式在接收阶段被捕获，而不是 MR 阶段。

**陷阱：**
- Requirement 永远"在进行中"，因为没人正式关闭——时间久了队列变得无意义。
- 把缺陷当成"开发团队的问题"，而不是通过 Defect Attribution 反馈，找出哪个 BA 阶段的缺口让它漏过。

## 按 BA 步骤的必需工件

| 步骤 | 必需工件 | Tier 差异 |
| --- | --- | --- |
| 1 接收 | BA 队列条目 | 同 |
| 2 分析 | [Feature Spec](../../../templates/feature-spec.md) 草稿 | Tier A：轻量版；Tier C：完整版 + ADR 预标 |
| 3 需求评审 | [Review Record](../../../templates/requirement-review-record.md) §1 | 同 |
| 4 技术评审 | [Review Record](../../../templates/requirement-review-record.md) §2，可选 [ADR](../../../templates/adr.md) | Tier C：有架构影响时必需 ADR |
| 5 测试评审 | [Review Record](../../../templates/requirement-review-record.md) §3 | Tier C：敏感时必需 Security Lead |
| 6 Story 拆分 | [Story Card](../../../templates/story-card.md) × N、追溯表 | 同 |
| 7 Backlog 入库 | [BA Handoff Checklist](../../../templates/ba-handoff-checklist.md)、[SDD Story Spec](../../../templates/sdd-story-spec.md)（Tier B/C） | Tier B/C 必需 SDD Story Spec |
| 8 Sprint 计划 | [Iteration Brief](../../../templates/iteration-brief.md) 更新 | 同 |
| 9 开发支持 | 更新的 Story Cards / SDD Story Specs / Requirement | 同 |
| 10 Story 验收 | 更新的 Requirement、Story Acceptance Records | 同 |
| 11 反馈闭环 | [Change Request](../../../templates/change-request.md)、[Knowledge Base Update](../../../templates/knowledge-base-update.md)、[Defect Attribution](../../../templates/defect-attribution.md)、更新的 Requirement | 同 |

## 每个 Requirement 的最小 BA 检查清单

声称 Requirement 已可拆分为 Stories 之前：

- [ ] Feature Spec 已 Approved（或 Conditionally Approved）。
- [ ] 三个评审节都完成，显式 Pass / Pass-with-conditions 结果。
- [ ] 每个"condition"都有 owner 和验证机制。
- [ ] Open question 列表为空。
- [ ] 至少一名 Tech Lead 和一名 QA 签了 §2 / §3。
- [ ] 受影响模块的 Module Owner 已知会。

- [ ] BA Handoff Checklist 对每个 Story 都通过。
- [ ] AI 可用性自测对每个 Story 都通过。
- [ ] 每个 Story 都记录 Tier 分类。
- [ ] Review Record 的 Traceability 表链接每个 Story。

关闭 Requirement 之前：

- [ ] 所有子 Story 已验收（或显式去除范围）。

- [ ] 反馈已分类和路由（缺陷 / 变更请求 / KB 更新）。
- [ ] 经验教训已记录。

## BA 工作如何支持 AI

每个 BA 产出的设计都是为了让下一步 AI 辅助不必猜：

- Feature Spec 成为所有派生 Story 的上游上下文。
- Review Record 成为审计轨迹——评审者问"这个范围哪儿来的"时 agent 能引用它。
- Handoff Checklist 的 AI 可用性自测是允许 AI agent 执行 Story 的前置条件。
- 开发期的澄清记录在 agent 会读的工件里，不在 agent 永远看不到的聊天里。
- 末尾的反馈成为下迭代 BA 工作能拾取的结构化输入——Change Request 和 Defect Attribution。

模式：**BA 产出 = agent 能读的文件。**任何只在 BA 头脑里或聊天里的东西，对下游 AI 工作不可见。

## BA 与执行栈的关系

| 栈层 | BA 的角色 |
| --- | --- |
| 1 SDD | 主要——BA 拥有规格层（Feature Spec、SDD Story Spec 输入、AC）。 |
| 2 Superpowers | 辅助——BA 通过确保 Story 真的 ready 来支持开发者的 `brainstorming`（DoR 检查）skill。 |
| 3 Harness | 间接——BA 定义 Harness 强制执行的 AI Context Boundary。 |
| 4 CI / Review | 间接——BA 参与 Story 验收并把验收回链到 Requirements。 |

当第 2-4 层出问题，Defect Attribution 追到"规格模糊"或"上下文缺失"时，这就是 BA 流程的信号——第 1 层输入不够强。

## 要点回顾

- BA 端到端拥有 S2 和 S3；矩阵里的其他位置 BA 是支持角色，但仍有落到工件里的具体产出。
- Feature Spec 是 BA 的中心工件；Review Record 是门禁证据；Handoff Checklist 是 AI 辅助 Story 执行的前置条件。
- "AI 可用性"是 BA 的责任——每个产出都得是 agent 能读、能用、不必再问 BA 的东西。
- Step 11 的反馈不是可选的；未关闭的 Requirement 污染队列、丢掉学习回路。
