# 术语表

英文版：[../../knowledge/12-glossary.md](../../knowledge/12-glossary.md)

## 目的

本术语表解释手册中使用的缩写和短术语。读者不应该因为不理解缩写而无法理解 AI-SDD 工作流。

## 核心术语

| 术语 | 全称 | 本手册中的含义 |
| --- | --- | --- |
| AI | Artificial Intelligence | 用于支持软件交付的 AI 工具、模型、agent 和 assistant。 |
| AI-SDD | AI-assisted Spec-Driven Development | 本手册中的治理模型：AI 辅助工作从批准规格开始，以基于证据的交付结束。 |
| SDLC | Software Development Life Cycle | 软件开发生命周期，覆盖从想法、需求、设计、实现、测试、发布、运行到持续改进的全过程。本手册中的 Team AI SDLC 指团队级 AI 辅助软件交付生命周期。 |
| SDD | Spec-Driven Development | 规格驱动开发，用规格、验收标准、约束和上下文边界驱动实现。 |
| Story | User Story | 描述用户价值、验收标准和实现范围的 待办列表 item。 |
| AC | Acceptance Criteria | 验收标准，即判断 Story 是否可接受的可测试条件。 |
| DoR | 就绪定义 | Story 进入开发前必须满足的最低条件。 |
| DoD | 完成定义 | Story 被认为完成前必须满足的最低条件。 |
| MR | Merge Request | GitLab 风格的代码合入请求。本手册默认使用 MR。 |
| PR | Pull Request | GitHub 风格的代码合入请求，许多流程中与 MR 等价。 |

## 架构与计划

| 术语 | 全称 | 本手册中的含义 |
| --- | --- | --- |
| ADR | Architecture Decision Record | 记录重要架构决策、背景、选项和后果的简短文档。 |
| API | Application Programming Interface | 服务、组件或平台暴露的接口契约。 |
| OpenAPI | OpenAPI Specification | 描述 HTTP API 的标准格式。 |
| JSON | JavaScript Object Notation | 用于 API、配置和事件载荷的结构化数据格式。 |
| YAML | YAML Ain't Markup Language | 常用于配置和政策文件的可读结构化数据格式。 |
| DDD | Domain-Driven Design | 围绕领域概念、边界和业务语言建模的软件设计方法。 |
| POC | Proof of Concept | 用于在大规模投入前验证可行性的小型实现。 |
| BMAD | Breakthrough Method of Agile AI-driven Development | 一个 AI 辅助敏捷框架，在本手册中作为上游发现或复杂工作升级路径参考。 |
| GSD | Get Shit Done | 一个 context engineering 和 spec-driven long-task execution system，在本手册中作为长任务 AI 执行模式参考。 |

| 术语 | 全称 | 本手册中的含义 |
| --- | --- | --- |

| CODEOWNERS | Code Owners File | 将仓库路径映射到责任 owner 或 评审er 的文件。 |

## 治理与角色

| 术语 | 全称 | 本手册中的含义 |
| --- | --- | --- |
| RACI | Responsible, Accountable, Consulted, Informed | 用于明确治理和交付活动角色的责任矩阵。 |
| BA | Business Analyst | 拥有上游 **需求 → 三道评审 → Story 拆分** 流程的角色。起草 Feature / Requirement Spec，主持需求评审，联合主持技术评审和测试评审，Backlog 入库前跑 BA Handoff Checklist，验收后关闭 Requirement。见 [BA 指南](../practice/09-ba指南.md) 和 [运行模型 § BA 问责](04-运行模型.md)。 |
| PO | Product Owner | 框定业务意图、定优先级、验收业务结果；与 BA 协作框定 Requirement。 |
| Owner Review | Owner Review | 由负责模块或领域的 accountable owner 对变更进行审批。 |
| AI Champion | AI Champion | 辅导 AI-SDD 使用、维护示例并收集失败案例的团队成员。 |

## 交付流程术语

这些术语描述把业务请求转化为 ready Story 的上游、客户驱动的流程。

| 术语 | 本手册中的含义 |
| --- | --- |
| Requirement（需求） | 客户或业务提出的 Feature 量级业务单元，记录在 [Feature Spec](../../../templates/feature-spec.md)（在此流程中作为 Requirement Spec 使用）。一个 Requirement 通常派生多个 Story。 |
| 需求评审 | 三道门禁评审中的第一道。确认*什么*：范围、成功标准、stakeholders。BA 主持。记录在 [Review Record](../../../templates/requirement-review-record.md) §1。 |
| 技术评审 | 三道门禁评审中的第二道。确认*怎么*：可行性、架构影响、依赖、风险。Tech Lead 主持。记录在 Review Record §2。 |
| 测试评审 | 三道门禁评审中的第三道。确认*可测试性*：AC 质量、必需测试层级、UAT 范围。QA 主持。记录在 Review Record §3。 |
| 三道评审 | 需求评审 / 技术评审 / 测试评审 的合称。Requirement 在整体 gate 决定是 *Approved* 或 *Conditionally Approved* 之前不能被拆成 Stories。 |
| BA Handoff Checklist | Story 进 Backlog 前的门禁（清单的 AI 可用性自测不通过的 Story 不进 Backlog）。见 [模板](../../../templates/ba-handoff-checklist.md)。 |
| Backlog | 已通过 BA Handoff Checklist、等待被 sprint 选中的 Story 队列。状态在 Jira / ONES / 类似工具里，本手册不为它建模。 |
| Sprint 选取 | 从 Backlog 选 Story 进具体迭代的动作。确认范围记录在 [Iteration Brief](../../../templates/iteration-brief.md)。 |
| Sprint | 一个交付迭代。Story 范围在 Planning 时冻结；中途变更走 Change Request。 |
| DoR Check | Definition of Ready 检查。双层：BA 在 Handoff Checklist 上的 AI 可用性自测（Backlog 前的门禁），加开发者自己的 DoR 检查（AI 执行前的门禁）。 |

## 优先级标签

| 术语 | 含义 |
| --- | --- |
| P0 | 受控试点的 必需。 |
| P1 | 团队扩展的 应具备。 |
| P2 | baseline 稳定后的 可选增强 或规模化能力。 |
| Tier A | 轻量、低风险变更。 |
| Tier B | 标准 Story 或有意义的业务变更。 |
| Tier C | 高风险、核心、生产影响、架构、权限、数据或安全敏感变更。 |
