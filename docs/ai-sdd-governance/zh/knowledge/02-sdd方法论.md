# SDD 方法论

英文版：[../../knowledge/02-sdd-methodology.md](../../knowledge/02-sdd-methodology.md)

## 定义

SDD 表示 Spec-Driven Development。

AI 辅助开发必须从经过评审的规格开始。开发者不应从自由 prompt 或直接代码生成开始。

## 端到端流程

1. 业务需求进入 待办列表。
2. 产品负责人 和团队将其拆分为 Epic、Feature、User Story 和 Task。
3. 每个 Story 获得 SDD Story 规格。
4. 技术风险需要 技术规格 或 ADR。
5. AI 基于批准的规格草拟代码、测试、文档和评审清单。
6. 开发者检查、调整并对输出负责。
7. MR 通过自动化质量门禁和人工评审。
8. 验收结果回流到规格、测试和知识库。

## 就绪定义

Story 只有满足以下条件才 ready：

- 业务目标。
- 用户角色。
- 前置条件。
- 主流程。
- 替代流程和异常流程。
- Given/When/Then 格式的验收标准。
- API 契约或数据字段说明。
- 权限、审计、性能、兼容性和安全等非功能要求。
- AI 上下文边界，说明可使用哪些文档、API、代码和测试。

## 完成定义

Story 只有满足以下条件才 done：

- 代码通过编译、静态分析、单元测试、集成测试和安全扫描。
- MR 说明 AI 使用方式。
- SDD Spec、测试规格 和 API 文档已更新。
- 关键业务逻辑具备人工评审记录。
- 工件可以追溯到需求、规格版本、代码变更、测试结果和发布版本。

## SDD 工件生命周期

Draft：

- 可由 产品负责人、BA、开发者或 AI Assistant 创建。
- 可以包含未解决问题。

Ready for 评审：

- 包含完整 DoR 信息。
- 有明确验收标准和 AI 上下文边界。

Approved：

- 由 产品负责人 和 Technical Owner 评审。
- 可作为 AI 输入。

Implemented：

- 链接到 MR 和测试证据。

Accepted：

- 由 QA 或业务代表验证。
- 经验教训加入知识库。

## AI 使用规则

允许：

- 从批准业务笔记草拟 SDD Spec。
- 从批准规格生成第一版代码。
- 生成单元测试、契约测试和边界场景列表。
- 总结缺陷模式和评审发现。
- 草拟文档和发布说明。

受限：

- 未经人工评审生成生产影响代码。
- 在 prompt 中使用未批准客户数据。
- 使用未脱敏生产日志。
- 未经 ADR 修改架构决策。
- 绕过 CI/CD 或 MR 评审。
